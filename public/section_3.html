Designing a REST API

REST APIs are one of the most common kinds of web services available today. They allow various clients including browser apps to communicate with a server via the REST API.

Accept and respond with JSON

REST APIs should accept JSON for request payload and also send responses to JSON. JSON is the standard for transferring data. JavaScript has built-in methods to encode and decode JSON either through the Fetch API or another HTTP client. We should also make sure that our endpoints return JSON as a response.

Using nouns in endpoint paths

We shouldn’t use verbs in our endpoint paths. Instead, we should use the nouns(in plural form) which represent the entity that the endpoint that we’re retrieving or manipulating as the pathname.

This is because our HTTP request method already has the verb. Having verbs in our API endpoint paths isn’t useful and it makes it unnecessarily long since it doesn’t convey any new information.

The action should be indicated by the HTTP request method that we’re making. The most common methods include GET, POST, PUT, PATCH and DELETE.

GET retrieves resources. POST submits new data to the server. PUT updates existing data in its entirety while PATCH updates parts of existing data. DELETE removes data. The verbs map to the CRUD operations.

API table

Let’s imagine we are building an API which handles data on music tracks. We can use the noun “tracks” for our endpoint. The following table describes the associations between endpoints, HTTP methods and the actions performed for each of them.

/tracks /tracks/id
GET Retrieve all existing tracks Retrieve track with specified id
POST Add one track
PATCH Update track with specified id
DELETE Delete track with specified id
